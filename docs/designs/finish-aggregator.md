## Goal

Bring the AI Tech News Aggregator to a state where every UI tab works with
real data and the whole thing runs on a multi-project VPS, polished enough
to share with backend / infra / platform engineering recruiters.

## Why now

The shell is functional, but four of the six tabs are either mocks
(Knowledge Graph), broken (Settings persistence), or quality-mediocre. The
recently-built skill suite (`test-app-e2e`, `grill-me`, `write-prd`,
`write-issue`, `missions`) is itself part of the portfolio narrative —
finishing this is the case study that ties the skills to a working
deployable product.

## Users

- **Primary**: the maintainer (single-user portfolio piece). Reads news
  every morning, occasionally points recruiters at the live URL.
- **Secondary**: backend / infra / platform-engineering recruiters and
  hiring managers. They will both click around the live demo *and* read
  the codebase. Code quality and architecture matter as much as UX.

## Scope

### In scope
- **Knowledge Graph (real)** — replace hardcoded mock with named-entity
  extraction (companies, people, products, technologies) from article
  text, persist entities + edges, expose `/api/knowledge-graph`, rewire
  the React component
- **Settings persistence (backend)** — add a settings store + endpoints
  (`GET/PUT /api/settings`); frontend reads on mount, writes on save;
  localStorage degrades to a cache, not the source of truth. *This was
  flagged as the worst-offender feature; gets prioritised.*
- **Stale test cleanup** — delete the 3 test files that import the
  removed `utils` package; they're noise the E2E suite already covers
- **Dead-code / repo polish** — pass through the codebase to remove
  leftover commented blocks, tighten docstrings, normalise patterns. The
  backend audience will read this.
- **Re-ingest from clean** — wipe `news.db`, re-ingest from RSS, re-run
  the orchestrator. Single-command path to a fresh, consistent data state.
- **Retention policy** — daily cron deletes articles where
  `published_at < now() - RETENTION_DAYS` (default 30); embeddings &
  entity-mentions cascade delete via FK
- **VPS deployment** — generic-Linux Docker Compose stack behind a
  shared Caddy reverse proxy; project lives at `news.<domain>.tld` once
  domain is chosen; co-tenants live at sibling subdomains
- **CI/CD** — push to `main` → GitHub Actions runs tests + build →
  pushes Docker image to GHCR → SSHes into the VPS →
  `docker compose pull && docker compose up -d`
- **Ollama on the VPS** — `llama3.2:1b` runs on the same box; budget
  ~2 GB RAM dedicated; Ollama loaded on demand with `keep_alive=5m` so
  it doesn't squat memory full-time

### Out of scope (explicit non-goals)
- Real auth / multi-user / login / users table
- Mobile-responsive design (desktop browser only)
- Migrating off SQLite to Postgres
- Hosted LLM providers (Groq, OpenAI, Anthropic) — Ollama only
- Real-time push / websockets — refresh-on-load is fine
- Uptime monitoring / alerting beyond what GitHub Actions tells us
- Replacing existing skills (`test-app-e2e`, `grill-me`, etc.) — extend,
  don't refactor
- Domain registration is parked — design works with raw IP first; domain
  swaps in later via DNS + Caddyfile edit

## Success criteria

1. Maintainer opens `http://localhost:3000` (and, eventually, the live
   subdomain), clicks every tab — News, Research, Chat, Digest, Knowledge
   Graph, Settings — and each shows real backend-driven content
2. The 13-test E2E suite stays green; new tests added for the new
   endpoints (`knowledge_graph_endpoint`, `settings_persistence`,
   `retention_job`)
3. Pushing to `main` triggers a deploy that ends with the live site
   reflecting the new HEAD within ~5 min, no manual SSH
4. The retention job runs daily and the article count stays bounded
   (no ever-growing disk usage)
5. A recruiter clicking the live URL sees a coherent product, and a
   recruiter reading the repo sees idiomatic, low-clutter code

## Constraints

- **Tech stack** (frozen): FastAPI, SQLAlchemy, SQLite, React + Vite,
  Tailwind + shadcn, Ollama, sentence-transformers (already installed)
- **Time**: spread across this week; autonomous mode — escalate only
  when validators fail 3× or scope shifts
- **Compatibility**: existing 13 E2E tests must stay green throughout
- **Compute**: Ollama (`llama3.2:1b`) must run on a VPS sized for
  multi-project hosting — total VPS minimum 4 GB RAM, 8 GB recommended.
  Hetzner CPX21 (3 vCPU / 4 GB / 80 GB / ~$8/mo) is the working
  reference spec
- **Cost**: free-tier-only LLM, no paid APIs; total VPS cost target
  $5–10/mo

## Data

- **Inputs**: same RSS feeds as today (5 sources, ~80 articles/run),
  plus user settings (categories, view prefs)
- **Outputs**: same SQLite DB as today, additionally
  `entities`/`entity_mentions`/`settings` tables
- **Source of truth**: `backend/news.db` (SQLite). Single store; no
  external state.
- **Lifecycle**: articles land via `/api/ingest/`, get summarised by
  the orchestrator, get entity-extracted into the graph tables, then
  fall off after 30 days via the retention cron
- **Persistence on deploy**: `news.db` is a Docker volume, survives
  container restarts and image updates; backed up via VPS-level
  snapshots (provider's tooling), no app-level backup script

## Failure modes & handling

| Risk | Mitigation |
|------|------------|
| Ollama hallucinates entities (single chars, common words, all-caps junk) | Reject extracted strings under 3 chars, all-uppercase non-acronyms, common stopwords; require minimum confidence; sanity check post-extraction |
| Re-ingest pulls partial RSS state on a bad-feed day | Tolerate; the orchestrator records sources_processed; if < 50% of feeds responded, log and skip retention until next ingest |
| Knowledge Graph canvas chokes on real-data scale (50+ nodes) | API limits to top-N entities by mention count; frontend adds zoom + filter controls |
| VPS RAM exhaustion when multiple projects + Ollama compete | Per-container memory limits in compose; Ollama with `keep_alive=5m` not perpetual; project-level OOM doesn't kill siblings |
| Retention job deletes the wrong rows | Dry-run mode for first prod run; log every deletion ID before commit; cap deletions per run at N to limit blast radius |
| CI/CD secrets leak | Never echo env vars in CI logs; use GitHub OIDC where supported; rotate deploy SSH key quarterly; least-privilege user on the VPS |
| Deploy breaks the VPS for other projects | Each project gets its own compose project name + network; OOM in one doesn't OOM siblings; failed deploy keeps the previous container running |
| Domain not chosen yet → can't get TLS via Let's Encrypt | Caddy supports raw-IP self-signed in dev; design assumes domain swap-in later via Caddyfile + DNS, no app changes |

## Alternatives considered (and rejected)

- **Hosted LLM providers (Groq / OpenAI / Anthropic)** — rejected: keeps
  it free + local; multi-provider abstraction is portfolio-worthy but
  was de-prioritised against deployment work.
- **Postgres migration** — rejected: SQLite is sufficient for single-user
  scale; the SQLAlchemy ORM means switching later costs ~1 hour.
- **Replace the KG tab with a simpler visualisation** — rejected: the
  user explicitly chose real entity extraction even at the cost of more
  ML wiring.
- **Run Ollama at home + tunnel to VPS** — rejected: more fragile (home
  internet flaps break prod), defeats the "runs on a public URL"
  portfolio claim.
- **Pre-summarise at home, ship a static DB to the VPS** — rejected:
  same fragility issue + the "live ingest" story is a portfolio asset.
- **Production deployment via Render/Vercel free tier** — rejected: the
  user wants a VPS that hosts multiple projects; Render free tier is
  per-project and wipes disk on deploy.
- **Retention via soft-delete + 90-day purge** — rejected for v1:
  simpler hard-delete suffices; no analytics use case yet.
