# Design context: Agentic Research Mode

> Generated by grill-me on 2026-05-08. Source of truth for the downstream
> PRD (`docs/prds/agentic-research.md`), the per-milestone issues
> (`docs/issues/agentic-research-*.md`), and the Mission run that will
> execute the work (`docs/missions/agentic-research.md`).

## Goal

Replace the misnamed "Agentic Research Mode" tab — currently a single
semantic-search call labelled as agentic — with a real multi-step LLM
agent. The agent decomposes a user question into 3-5 sub-questions, runs
internal semantic search per sub-question against `news.db`, synthesizes
findings into a structured markdown report, and streams the result to the
UI in real time.

## Why now

The current Research tab is the most visibly "fake" feature in the app.
Recruiter clicks it, types a question, sees `0 results / 0 sources used`.
Replacing it with a true agentic flow is the single highest-leverage
portfolio upgrade left — it demonstrates real LLM orchestration, SSE
streaming, multi-step reasoning, and structured output, all of which the
backend / infra audience actively cares about.

## Users

- **Primary**: the maintainer (single-user portfolio piece). Same as the
  rest of the app.
- **Secondary**: backend / infra / platform engineering recruiters and
  hiring managers. They will type a real research question, watch the
  agent work, and judge both the UX (streaming, phase progression) and
  the code (agent loop, error handling, citation rigor).

## Scope

### In scope
- New `AgenticResearchService` in `backend/src/services/` that runs the
  agent loop:
  1. **Decompose** — LLM call against `gpt-oss:20b` produces 3-5
     sub-questions in JSON
  2. **Search** — for each sub-question, call the existing semantic
     search (top-N hits, capped) against the local corpus
  3. **Synthesize** — final LLM call streams a structured markdown report
- New `POST /api/research` endpoint exposing the agent via Server-Sent
  Events (SSE). Each event carries either a phase update or a token chunk
  of the streaming report
- Frontend `ResearchMode.tsx` rewired to consume SSE: phase chip showing
  `Decomposing → Searching (2/4) → Synthesizing → Done`, report streaming
  in token-by-token below
- Numeric inline citations (`[1]`, `[2]`) anchored to a "Sources Used"
  section at the bottom; each source links back to the original article
- Best-effort failure handling — sub-questions that return zero hits
  don't fail the run; the report explicitly states "Could not find data
  on X"
- Single in-flight run per user; submit button disabled while running
- Export-only persistence — user can copy or download the markdown.
  No DB table, no sidebar of past reports
- New Playwright spec exercising the full SSE roundtrip + asserting the
  8-category UX rubric (no entities, real sources, citations resolve,
  no horizontal overflow on the streaming pane, etc.)

### Out of scope (explicit non-goals)
- External web search (DuckDuckGo, Brave, scraping)
- Persisting reports to a `research_reports` DB table
- Shareable public `/research/<slug>` URLs
- Tool-using agent (calling `/api/digest`, `/api/news/stats`, knowledge
  graph endpoints, etc.) — agent only sees the semantic-search corpus
- Iterative chat / follow-up refinement (one-shot per query)
- Quantitative table or chart output (markdown prose only)
- Configurable model per request — pinned to `gpt-oss:20b`
- Smart "trivial question" triage that skips decomposition — agent always
  decomposes, even on simple inputs
- Concurrent runs / job queue / background notification

## Success criteria

1. Maintainer types a research question, watches the phase chip advance,
   sees the report stream in token-by-token, and gets a structured
   markdown report with at least 3 sub-questions answered and at least
   1 inline citation that resolves to a real article URL
2. Total wall-clock under 5 minutes for a typical query against
   `gpt-oss:20b` on the user's CPU
3. Cancelling the tab mid-run terminates the Ollama generation server-side
4. New Playwright `research.spec.ts` test exercises the full SSE flow,
   passes the 8-category UX rubric, AND the existing 25-test suite stays
   green
5. The legacy `/api/search/semantic` user-facing call from the Research
   tab is gone; semantic search continues to exist as the agent's
   internal tool

## Constraints

- **Tech stack** (frozen): FastAPI, SQLAlchemy, SQLite, React + Vite,
  shadcn/ui, Ollama, sentence-transformers (already installed)
- **Model**: `gpt-oss:20b` for the agent (already pulled, 13 GB on disk).
  `llama3.2:1b` stays for the existing summarisation + RAG paths.
- **Time per query**: hard cap 5 minutes wall-clock; SSE keepalive every
  10s so the connection doesn't time out on a slow synthesize step
- **Compute**: single-user app, single in-flight run; Ollama serializes
  internally so no real concurrency benefit anyway
- **Compatibility**: must not break the 16-test contract suite or the
  25-test Playwright suite; existing `/api/search/semantic` endpoint
  stays alive as an internal tool / debugging surface

## Data

- **Inputs**: user's free-text research question; the existing 80-row
  `news.db` corpus
- **Outputs**: a streaming markdown report; the agent emits SSE events of
  shape `{type: "phase" | "token" | "done" | "error", data: ...}`
- **Source of truth**: `news.db` (no new tables, no migrations)
- **Persistence on session boundary**: nothing — reports live only in the
  user's browser memory (export-only by design)

## Failure modes & handling

| Risk | Mitigation |
|------|------------|
| Sub-question returns 0 hits | Best-effort: report says "Could not find data on X"; don't fail the run |
| Decomposer LLM emits invalid JSON | Strict parser + 1 retry with stricter prompt; if still bad, treat the original question as a single sub-question and proceed |
| `gpt-oss:20b` OOMs the user's machine | Catch the failure, return SSE error event with provider-fallback hint ("try llama3.2:1b"); do not retry automatically |
| Ollama crashes mid-stream | SSE closes cleanly with `error` event; frontend shows "research interrupted; retry?" — not a hang |
| User closes the tab mid-run | Backend detects disconnected SSE client, cancels the in-flight Ollama generation |
| Run exceeds 5-minute cap | Server-side timer aborts the agent, sends final SSE event with the partial report we have so far |
| Two concurrent submits | Frontend disables submit while running; backend rejects with 429 if the gate slips |
| Citations don't anchor (broken markdown links) | Render markdown via a strict renderer; Playwright test asserts each `[N]` link resolves to a `#source-N` element in the DOM |

## Alternatives considered (and rejected)

- **Single enriched search with structured output** — rejected: same UX
  as today, just dressed up. The "agentic" claim should be backed by real
  multi-step behaviour
- **Full tool-using agent (digest, knowledge graph, stats)** — rejected
  for v1: too many failure surfaces; debug story gets complicated; not
  necessary to demonstrate the pattern
- **Iterative chat with research mode toggle (Perplexity-style)** —
  rejected: stateful conversation adds session/history concerns we
  explicitly want to skip in scope
- **External web search (DuckDuckGo / Brave)** — rejected: introduces a
  vendor dependency, rate-limit story, and changes the "purely local"
  demo claim. Internal corpus is enough for portfolio
- **Persist reports to DB with a sidebar** — rejected for v1: simpler
  export-only path lands faster; can be added later
- **`llama3.2:1b` for the agent** — rejected: 1B model fails JSON-output
  assertions and can't reliably decompose questions. `gpt-oss:20b` is
  already pulled and available
- **Smart "trivial question" triage that skips decomposition** —
  rejected: misclassifications hide the agent behaviour; portfolio
  story benefits from showing the work even on simple questions

## Open questions (parked, will resurface during PRD)

- Exact phase-chip wording — "Decomposing" vs "Planning" vs "Sketching"
  vs an icon; bikeshed-able
- Whether the Sources section shows article snippets inline or just
  titles + source + link
- How to test SSE in Playwright (Playwright has experimental SSE; may
  need to fall back to fetch + manual stream parsing)
- Whether to expose model choice as a query param even though we're
  committing to `gpt-oss:20b` — gives a config knob for future tuning
- Whether the decomposer should see the article corpus's available
  sources / categories / date range as context (improves sub-question
  quality but adds prompt complexity)

## Conversation log

### Round 1 — agent shape & output
- **What does "Agentic" mean?** → Multi-step LLM agent (decompose → search → synthesize)
- **Output report shape?** → Structured sections (Executive Summary | Key Findings | Trends & Themes | Sources | timestamp)
- **Time budget per query?** → 5-min cap with SSE progress streaming
- **Persistence?** → Export-only (markdown copy/download), no DB

### Round 2 — operational
- **Search source?** → Internal-only (semantic search over news.db)
- **Live UX?** → Phase indicator + token streaming
- **Citations?** → Numeric inline + sources section anchored by `[N]`
- **Failure mode?** → Best-effort with explicit gaps

### Round 3 — model & API
- **Model?** → `gpt-oss:20b` (already pulled)
- **Decomposition strategy?** → Always decompose into 3-5 sub-questions
- **API surface?** → Replace the current `/api/search/semantic` call from the Research tab; semantic search becomes an internal tool
- **Concurrency?** → Max 1 research run at a time per user
