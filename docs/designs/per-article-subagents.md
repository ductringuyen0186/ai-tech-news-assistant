# Design context: Per-article subagents + deepagents SDK

> Generated by grill-me on 2026-05-08. Source of truth for the downstream
> PRD (`docs/prds/per-article-subagents.md`), per-milestone issues
> (`docs/issues/per-article-subagents-*.md`), and the Mission run that
> will execute the work (`docs/missions/per-article-subagents.md`).

## Goal

Refactor the agent loop to **isolate per-article reasoning into its own
subagent context**. The orchestrator agent never sees raw article body
text — only IDs and per-article summaries returned by subagents. A
`max_concurrent = 4` semaphore-bounded pool runs subagent work in
parallel against Ollama. The langchain **deepagents** SDK provides the
subagent runtime + skill registry; four skills (`search_articles`,
`summarize_article`, `extract_entities`, `query_knowledge_graph`) are
exposed to the agent and chosen dynamically.

This **replaces M1's `AgenticResearchService`** end-to-end, preserving
the public SSE event shape from previous Mission 1 (M1+M2+M3+M4+M5)
while augmenting it with per-subagent telemetry.

## Why now

Three motivations stack:

1. **Token-window control.** With many articles in a single synthesis
   prompt, `gpt-oss:20b`'s context fills up and either degrades or OOMs.
   Per-article subagents bound each context to one article + the user's
   question.
2. **Quality of synthesis.** Forcing one-article-at-a-time analysis
   produces tighter per-article reasoning that the orchestrator
   aggregates — a divide-and-conquer pattern that's measurably cleaner
   than throwing everything into one prompt.
3. **Speed via concurrency.** `max_concurrent = 4` subagents in flight
   against Ollama (or against an external LLM if configured) cuts
   wall-clock significantly on bulk-article queries.

The deepagents SDK earns the dependency by giving us:
- A canonical agent loop that knows how to dispatch subagents
- A skill/tool registry pattern we'd otherwise build by hand
- Context-engineering hooks (filter what the orchestrator sees per call)
- Battle-tested error handling for failed subagent invocations

## Users

- **Primary**: the maintainer (single-user portfolio piece). The
  end-to-end UX from Mission 1 stays the same — type a question, watch
  phases stream, read the report.
- **Secondary**: backend / infra / platform engineering recruiters who
  click into the source code or the Playwright video and see real
  multi-agent orchestration in action. The full subagent event stream
  surfaces to the frontend, making the architecture *visible*, which is
  the entire portfolio point.

## Scope

### In scope

- **Adopt deepagents SDK** (`pip install deepagents` or equivalent
  langchain-deepagents package) as the agent runtime
- **4 skills** implemented as deepagents tools, each wrapping an
  existing service:
  - `search_articles(query, top_k=5)` → `SearchService.search`
  - `summarize_article(article_id, focus_question=None)` →
    `SummarizationService` *with cache reuse* (read from
    `news.db.summary` if fresh, else generate)
  - `extract_entities(article_id)` → `EntityExtractionService.process_article`
  - `query_knowledge_graph(entity_or_topic)` → walks the M6
    entity/co-mention graph
- **`SubagentPool`** with `asyncio.Semaphore(4)` cap, configurable via
  `MAX_CONCURRENT_SUBAGENTS` env var
- **Per-article subagent contexts** — each summarize/extract call runs
  in its own deepagents subagent with ONLY the article body and the
  user's question in scope. Orchestrator sees only the *output*
  summaries
- **Replace M1's `AgenticResearchService`** end-to-end. Same public
  endpoint (`POST /api/research`), same outer SSE phase events
  (`Decomposing → Searching → Synthesizing → Done`), but augmented
  with per-subagent events
- **Subagent telemetry events** added to the SSE stream:
  ```json
  {"type": "subagent", "data": "start", "skill": "summarize_article", "article_id": 42}
  {"type": "subagent", "data": "done",  "skill": "summarize_article", "article_id": 42, "duration_ms": 4321}
  {"type": "subagent", "data": "error", "skill": "summarize_article", "article_id": 42, "message": "..."}
  ```
- **Frontend renders subagent activity** — `ResearchMode.tsx` gets a
  collapsible "Subagents" panel showing per-article progress
- **Best-effort failure semantics** — a failed subagent never aborts
  the run; the orchestrator continues with the others and the report
  notes "Could not analyze article #X"
- **Cache reuse** — `summarize_article` checks `news.db.summary` and
  reuses if (a) it exists and (b) the focus_question is null or
  matches a prior question (heuristic: same first-100-chars). This
  saves ~70% of per-article work on hot articles
- **Ingestion-time path benefits** — the summarization
  pipeline (`SummarizationOrchestrator`) is migrated to use the same
  `summarize_article` skill so there's one summarization implementation
  for both ingestion and research
- **Playwright + unit + integration tests updated** to cover the new
  event shape, the subagent panel UI, and the cache-reuse fast path

### Out of scope (explicit non-goals)

- **GPU / multi-Ollama-instance support** — `max_concurrent = 4` against
  a single Ollama instance is the target. Multi-instance load balancing
  is a v2 concern.
- **External LLM provider switch** — still pinned to Ollama
  `gpt-oss:20b`. deepagents supports OpenAI/Anthropic but we're not
  introducing those dependencies in this mission.
- **Per-question summary store** — the cache layer is "reuse if fresh",
  not a sophisticated per-question summary table. v2 if needed.
- **Subagent retry policy** — best-effort means one attempt per
  subagent, no retry. v2 can add `retry_once` if logs show flakiness.
- **Configurable per-skill failure policy** — flat best-effort across
  all skills.
- **fetch_full_article and list_sources skills** — out of M1's 6
  candidates, we picked 4. The other two land in v2 if needed.
- **Streaming subagent token output** — subagents return their
  finished summary; tokens within the subagent are not streamed up
  to the frontend (that would defeat the context-isolation point).
- **Frontend redesign of the report area** — the existing report card
  + phase chip stays. New "Subagents" panel is *additive*, not a
  replacement.
- **Background job queue** — subagents run inside the request lifecycle
  (just like M1+M2). No Celery, no separate worker.

## Success criteria

1. The user types a research question, watches the phase chip cycle,
   AND watches a "Subagents" panel light up with per-article progress
   (start/done events). The full report still streams in token-by-token
   below.
2. `MAX_CONCURRENT_SUBAGENTS=4` is honored — at most 4 subagent events
   are in `start` state at any given time during a run.
3. Per-article subagent context is bounded: a subagent processing
   article 42 has ONLY article 42's body + the user's question in its
   prompt. Verifiable via logs (each subagent's prompt size is
   roughly constant regardless of how many other articles exist).
4. Cache reuse: for articles that have a fresh stored summary in
   `news.db.summary`, the `summarize_article` skill returns it without
   calling Ollama. Verifiable via logs (`cache_hit=true`) and via wall
   clock improving on warm runs.
5. Best-effort failures: forcing one subagent to throw (via test stub)
   does NOT abort the run — the report includes the remaining articles
   and explicitly mentions the failed one.
6. Existing 5 Playwright tests in `research.spec.ts` (mocked) stay
   green. The live integration test (`research.live.spec.ts`) stays
   green. Add a new test asserting subagent events appear in the SSE
   stream.
7. Wall-clock under 5 minutes for typical queries on the maintainer's
   CPU — concurrency makes this easier to hit, not harder.
8. Ingestion summarization continues to work — articles still get
   summarized at ingest time, but now via the same `summarize_article`
   skill code path used by the research agent.

## Constraints

- **Tech stack** (frozen): FastAPI, SQLAlchemy, SQLite, React + Vite,
  shadcn/ui, Ollama, sentence-transformers — all unchanged
- **New dependency**: `deepagents` (or whatever the langchain package
  name resolves to). Pinned version recorded in `backend/requirements.txt`
- **Model**: `gpt-oss:20b` for both orchestrator and subagents. The 13GB
  weights are already on disk
- **Concurrency cap**: 4. Configurable via `MAX_CONCURRENT_SUBAGENTS`
  env var, default 4
- **Compatibility**: existing 17 contract tests + 29 Playwright tests
  must stay green (with research-spec tests possibly modified to assert
  new subagent events)
- **Public API stability**: `POST /api/research` keeps same JSON request
  body and same outer phase events. Subagent events are additive
- **Time budget**: 1 week to ship M1 through M6

## Data

- **Inputs**: user research question; existing `news.db` corpus;
  `news.db.summary` cache (already populated at ingest time)
- **Outputs**: streaming markdown report (unchanged); enriched SSE
  event stream that now includes `subagent` lifecycle events
- **Source of truth**: `news.db` (no new tables; summary cache lives
  in the existing `articles.summary` column)
- **New persistence layer**: NONE. Subagent state lives only in
  process memory during a run

## Failure modes & handling

| Risk | Mitigation |
|------|------------|
| `deepagents` SDK doesn't fit our async-generator pattern | M1 is a spike — if the SDK can't drive an async generator cleanly, we fall back to a custom `SubagentPool` + asyncio.Semaphore implementation. PRD's M1 has explicit go/no-go gates |
| Subagent fails (Ollama timeout, malformed output, etc.) | Best-effort: catch the exception inside the subagent dispatch, emit `subagent: error` event, continue the run with the remaining subagents |
| `max_concurrent = 4` thrashes Ollama | Tune via env var. PRD ships with default 4, monitoring logs for queue starvation |
| Cache-reuse returns a stale or off-topic summary | The cache key is `(article_id, focus_question_hash)`. If `focus_question` differs significantly from the cached question, regenerate. Heuristic in M2 |
| Frontend's existing tests break on the expanded SSE shape | Mock SSE in `research.spec.ts` already filters unknown event types — no breakage. Live tests get explicit subagent assertions |
| Per-article context still leaks to orchestrator via the LLM's tool-use payload | deepagents has explicit context-filtering hooks; M3 wires them. Verified in unit tests by inspecting the orchestrator's prompt size |
| `gpt-oss:20b` produces invalid JSON for the agent's tool-call decision | deepagents has its own JSON-output retry logic; we lean on it. Fall through to a single-question mode if the agent crashes (matches M1's existing pattern) |
| Migration of ingestion summarization breaks the existing pipeline | Ship behind a feature flag the first run; cut over once green |
| The deepagents Agent's prompts blow context on the *orchestrator* side | Aggressive context filtering: orchestrator sees ONLY article IDs + per-article summaries. No raw text. Verified via M3 unit tests |

## Alternatives considered (and rejected)

- **Custom asyncio.Semaphore + dataclass-based skill registry** —
  rejected (for the primary path; kept as M1 fallback). The user
  explicitly wants to evaluate deepagents and earn the SDK
  experience. If M1 spike fails, we revert to this.
- **Keep M1 entirely, only add caching** — rejected: doesn't address
  the context-isolation goal. Caching alone doesn't bound the
  orchestrator's prompt size.
- **Two-skill split (summarize_article_generic vs
  summarize_article_focused)** — rejected as too much surface area
  for v1. One `summarize_article(article_id, focus_question=None)`
  with cache reuse covers both cases.
- **Add /api/research/v2 alongside v1** — rejected as too much
  endpoint sprawl. User picked "Replace M1 entirely" because the SSE
  shape only *grows* (new event type), so old clients don't break.
- **Configurable per-skill failure policy** — rejected as v2. Flat
  best-effort is enough now.
- **External LLM provider via deepagents** (OpenAI / Anthropic) —
  rejected: stays Ollama-only for the local-only demo claim.
- **Streaming token output from subagents up to the frontend** —
  rejected: defeats the context-isolation point. Subagents return
  their finished summary; only the *orchestrator's* synthesis tokens
  stream up.
